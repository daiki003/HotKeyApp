# HotKeyApp Project Rules

このファイルは HotKeyApp リポジトリで作業するエージェント向けのプロジェクト固有ルールです。

## UI ルール

- 設定画面などで「特定の値 2 つを 1 組で登録する」UIは、`Views/GitSettingsDialog.xaml` の Git 設定エイリアス画面と同じパターンに統一する。
  - 標準パターンは次の構成とする。
    - 左端にドラッグハンドル `☰`
    - 2 つの入力欄
    - 右端にインラインの `削除` ボタン
    - 下部に `新規追加` ボタン
  - `エイリアス型設定画面` という用語は、行の見た目や入力欄だけでなく、一覧の上部に置く内部タブやその切り替え UI を含む共通部分全体を指す。
  - 今後 `エイリアス型設定画面` の改修と言った場合は、`エイリアス` 画面だけの個別改修として扱わず、`リポジトリ名`、`定数定義`、`選択式テンプレート` など同系統画面へ横展開すべき共通部分を先に検討する。
  - `エイリアス` 画面だけの内部タブ、タブ名編集、タブ追加、タブ切り替え挙動などを個別に指したい場合は、`エイリアス型設定画面` ではなく `エイリアス設定画面` と呼ぶこと。
  - 操作感もそろえる。
    - ドラッグ＆ドロップで並び替え可能
    - `Shift+Up` / `Shift+Down` で行の並び替え可能
    - 矢印キーで隣接セルや前後行へフォーカス移動できる
  - 今後 `定数定義`、`選択式テンプレート`、その他のキーと値のペア登録画面を追加する場合も、この標準パターンを優先する。
  - このプロジェクトでは、この標準パターンを「エイリアス型設定画面」と呼ぶこととする。
  - `エイリアス型設定画面` に対する見た目やキーボード操作の改修は、`Git 設定 > エイリアス` だけで閉じず、同じ設計の `リポジトリ名`、`定数定義`、`選択式テンプレート` にも原則として同時反映する。
  - 具体的には、フォーカス時に高さが変わらないこと、デフォルトの点線フォーカス枠を出さないこと、`Home` / `End` などの移動ルール、列見出しの表示有無など、共通化できる振る舞いは横並びでそろえる。
  - F2 などで表示用テキストをインライン編集用 `TextBox` に切り替える UI では、編集開始前後でボタンや行の高さを変えないことを絶対ルールとする。見た目だけでなく、余白・`Padding`・`MinHeight` を含めて通常表示と同等の高さを維持する。

## 実装方針

- 2 項目ペア登録 UI を新設するときは、既存の `SettingsWindow` 内で個別実装を増やす前に、Git 設定エイリアス画面との差分が最小になるように見た目とイベント構成を揃える。
- ラベル名だけが違う画面では、列構成、余白、ボタンスタイル、入力欄スタイルをできるだけ共通化する。

## ビルド環境の整合性維持

- **WPF一時ファイルのクリーンアップ**:
  WPFのビルドプロセス中に `*_wpftmp.csproj` がルートディレクトリに生成されることがあります。ビルドの強制終了やロックなどによりこれらが残留すると、`error MSB1011` (複数のプロジェクトファイルが存在する) が発生しビルドに失敗します。
  - **対応策**: `dotnet build` を実行する前に、ルートに `*_wpftmp.csproj` が存在しないか必ず確認してください。もし存在する場合は、ビルド前に即座に削除してください。
- **ログファイルの整理**:
  ビルドログ（`build_*.txt`、`*.log`）などが大量に残るとディレクトリが煩雑になります。調査が完了したログファイルは適宜削除し、ディレクトリをクリーンに保ってください。

## 作業進行のルール

- **作業完了後の必須手順**:
  コードの修正やビルド確認が終わった後は、必ず以下の手順を**一連の動作として**完了させてください。
  1. `taskkill /f /im HotKeyCommandApp.exe` で実行中のアプリを強制終了する。
  2. `dotnet build` でビルドを通す。
  3. `dotnet run` でアプリを起動し、ユーザーがすぐに確認できる状態にする。
- **`dotnet run` 実行時の判断基準**:
  WPF アプリでは `dotnet run` 実行後、アプリが正常起動していてもコマンド自体は終了せず待機し続けることがあります。Codex から実行した場合、一定時間後にタイムアウト表示になっても、それだけで起動失敗とは判断しないでください。
  - **確認方法**: タイムアウト後は `Get-Process HotKeyCommandApp` などで対象プロセスの有無を確認し、プロセスが存在すれば「起動成功」と扱ってください。
  - **運用方針**: `dotnet run` の終了可否ではなく、ウィンドウ表示または `HotKeyCommandApp.exe` プロセスの起動確認をもって成否判定を行ってください。
- 修正後にテストビルドしてくれる際、基本的にアプリを開いている状態なのでビルドが失敗すると思います。
  その際、即アプリを強制終了していいので、ロックされたまま何度も繰り返して時間を無駄にすることはしないでください。
